**各层之间的核心逻辑连接**：

- **世界观 → 基础**：知道 LLM 是什么（有损压缩、上下文窗口），才能理解为什么要从最简单的形式开始构建
- **基础 → 架构**：能构建和调试基础模型，才能理解注意力机制和 Transformer 每个组件的存在理由
- **架构 → 训练流程**：理解了架构，才能理解三阶段训练如何把架构变成 ChatGPT，分词如何影响一切
- **训练流程 → 规模与涌现**：理解了单个模型的训练，才能理解规模定律和 RL 训练中的涌现现象
- **规模与涌现 → 应用**：理解了 LLM 的全貌，才能在应用层做出合理的工具选择和安全判断

## 场景应用地图

### 场景 1：评估一个 LLM 输出，决定是否可以信任

**相关框架**：第2章（有损压缩）→ 第3章（上下文窗口）→ 第13章（参差不齐的前沿）

**执行顺序**：
1. 判断这个问题属于高频知识还是低频知识（第2章）
   - 高频（通用代码语法、广为人知的历史）→ 可信度高
   - 低频（最新事件、专有信息、罕见领域）→ 需要核实
2. 检查上下文：是否把相关原文放进了对话？（第3章）
   - 是 → 模型在"读"而非"回忆"，可靠性更高
   - 否 → 依赖参数记忆，有损压缩，精确度下降
3. 评估任务类型（第13章）
   - 综合分析、文本改写、代码解释 → 通常可靠
   - 精确事实、具体数字、来源引用 → 必须核实

---

### 场景 2：为专业任务配置最优 LLM 工作流

**相关框架**：第3章（上下文窗口）→ 第8章（三阶段）→ 第12章（LLM OS）

**执行顺序**：
1. 明确任务的信息需求（第3章）
   - 需要最新信息？→ 使用带搜索工具的模型（Perplexity / ChatGPT 搜索模式）
   - 需要处理专有文档？→ 上传原文到上下文
   - 需要执行计算？→ Code Interpreter
2. 理解哪个训练阶段的能力在起作用（第8章）
   - 世界知识问题 → 依赖预训练
   - 格式和风格问题 → 依赖 SFT
   - 多步推理问题 → 选择推理模型（o1, o3, Claude 推理模式）
3. 设计智能体循环（第12章）
   - 简单任务 → 单次对话
   - 研究整合 → Deep Research 模式
   - 多步自动化 → 智能体循环 + 工具调用

---

### 场景 3：调试 LLM 给出了糟糕的答案

**相关框架**：第2章（有损压缩）→ 第3章（上下文窗口）→ 第9章（分词）

**执行顺序**：
1. 问题是关于最新事件或专有信息吗？→ 知识截止或知识压缩问题（第2章）
   - 解决方案：直接提供相关资料
2. 问题涉及字符级操作（数字母、精确计数）吗？→ 分词问题（第9章）
   - 解决方案：格式化输入（用分隔符分开字符），使用 Code Interpreter 做精确计算
3. 模型没有遵循格式要求？→ 提示词问题（第3章）
   - 解决方案：改为少样本提示（提供 2-3 个示例），而非描述要求

---

### 场景 4：评估和选择 LLM 工具

**相关框架**：第10章（规模定律）→ 第11章（涌现）→ 第13章（前沿）

**执行顺序**：
1. 看参数量 + 训练 token 数（第10章）
   - 小模型 + 充分训练 vs 大模型 + 欠训练，前者在很多任务上更好
   - Chinchilla 比率：每个参数约 20 个训练 token 为最优
2. 任务需要推理能力吗？（第11章）
   - 需要多步数学/逻辑推理 → 优先选推理模型（经过 RL 训练）
   - 普通问答和写作 → 普通聊天模型即可
3. 在你的具体任务上测试，不要外推（第13章）
   - 边界是参差不齐的，不要因为模型在任务 A 表现好就假设任务 B 也好

## 常见困境解决框架

### 困境 1："学了忘用"——知道框架但实际用 LLM 时还是凭直觉

**根本原因**：框架还停留在描述层面，没有内化为习惯。

**解决方法**：
- 为每个框架设计一个"触发词"。比如，每次你问 LLM 一个精确事实时，自动触发"有损压缩"框架，问自己：这是高频还是低频信息？
- 在你的提示词里加上一个显性的自检步骤，强迫自己过一遍框架，直到它变成肌肉记忆

### 困境 2："用了没效果"——按照书里说的做，但 LLM 输出还是很差

**根本原因**：框架是对的，但应用场景判断错了，或执行细节有偏差。

**解决方法**：
- 用第13章的"参差不齐的前沿"框架：你的任务可能恰好落在当前 LLM 的能力盲区
- 对比不同模型在同一任务上的输出——有时候是模型问题，不是方法问题
- 降低抽象层次：如果"上下文窗口"方法没效果，把粘贴的内容再具体化、更直接

### 困境 3："改变了但别人没变"——你用 LLM 提效了，但团队其他人没有跟上

**根本原因**：LLM 工作流的效益需要一定上手成本，难以"只看结果"就说服别人。

**解决方法**：
- 用具体数字说话：记录你完成特定任务的时间，对比使用前后
- 从最简单、最低风险的任务开始推广（查信息、总结文档、格式转换），而不是从高风险决策开始
- 展示第13章的框架——哪些任务 LLM 可靠，哪些需要核实。给同事一个安全感，告诉他们"我们用 LLM 做什么、不做什么"
